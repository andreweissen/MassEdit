(readonly) Buttons :object

The Buttons pseudo-enum stores all seven config objects used to assemble ModalButton instances during the Modal creation process. Prior to UCP update 3, this data was stored in a static fashion within main.buildModal. However, due to the need to maintain a static default disabled flag for use in toggling buttons elements within the body of main.toggleModalComponentsDisable, the associated config objects were removed from the buildModal method and provided their own dedicated pseudo-enum. The modal builder function now uses this data to dynamically assemble the buttons and modal in an automatic fashion.

The key for each Buttons object entry is as follows: 
- HANDLER: Click handler function, a property of MassEdit instance
- TEXT: i18n message name for button text
- EVENT: Name of related click event (should be same as object key name)
- PRIMARY: boolean flag for primary styling
- DISABLED: boolean flag for default disabled behavior
- ID: Name of Selectors key related to element id
- CLASSES: Array of Selectors keys for class selectors

(readonly) Dependencies :object

This pseudo-enum of the init namespace object used to initialize the script stores data related to the external dependencies and core modules required by the script. It consists of two properties. The former, a constant object called "ARTICLES," originally contained key/value pairs wherein the key was the specific name of the mw.hook and the value was the script's location for use by importArticles.articles. However, this system was eventually replaced in favor of an array of objects containing properties for hook, window.dev alias, and script for more efficient, readable loading of dependencies.

The latter array, a constant array named MODULES, contains a listing of the core modules required for use by mw.loader.using.

The key for the ARTICLES array entries is as follows: 
- DEV/WINDOW: The name and location of the window property
- HOOK: The name of the mw.hook event
- ARTICLE: The location of the script or stylesheet on the Dev wiki
- TYPE: Either "script" for JS scripts or "style" for CSS stylesheets

(readonly) Flags :boolean

This pseudo-enum replaces the previous pair of boolean constants housed in the script-global execution context, namely DEBUG and TESTING. This enum houses the default values of these flags, the value of which are applied to the properties of the MassEdit instance's flags local variable. This system allows for the exposing of certain public methods that permit post-load toggling of the debug and test modes for more dynamic debugging.

(readonly) Globals :object

This pseudo-enum is used to store the string names of the various WikipediaGlobal (wg) variables required by the main MassEdit class instance and init object. These are fetched within the body of the init.preload function via a mw.config.get invocation and stored in an instance variable property named globals for subsequent usage. This approach replaces the deprecated approach previously used in the script of assuming the relevant wg variables exist as properties of the window object, an assumption that is discouraged in more recent version of MediaWiki.

(readonly) Scenes :object

The Scenes pseudo-enum is used to store data and names related to building the four major operations supported by the MassEdit script, namely find-and-replace, append/prepend content, message users, and generation of page listings. Originally, this enum was an array used to store the string names of the four main scenes in the order that determined their placement in the associated operation dropdown menu. The actual design schema used to dynamically build the string HTML from builder functions was stored within a since-removed method called main.buildModalScenes that built all the scenes at once on program initialization.

However, with the decision to revise this messy approach in favor of lazy-building scenes only when requested by the user, the scene schema was moved to this enum and rearranged into object form. The author was forced to make use of many nested Object.freeze invocations due to the inability to deep-freeze the whole object, though alternate approaches are being researched to de-uglify the enum in a future update.

(readonly) Selectors :string

This pseudo-enum of the main namespace object is used to store all CSS selectors in a single place in the event that one or more need to be changed. The formatting of the object literal key naming is type (id or class), location (placement, modal, content, preview), and either the name for ids or the type of element (div, span, etc.). Originally, these were all divided into nested object literals as seen in Message.js. However, this system became too unreadable in the body of the script, necessitating a simpler system.

(readonly) UserGroups :Array.<string>

This pseudo-enum of the main namespace object is used to store a pair of arrays denoting which user groups are permitted to make use of the editing and messaging functionality (all users are permitted to generate lists). For the purposes of forstalling the use of the script for vandalism or spam, its use is limited to certain members of local staff, various global groups, and Fandom Staff. The only major local group prevented from using the editing function is the threadmoderator group, as these can be viewed as standard users with /d and thread-specific abilities. However, these users are permitted to make use of the mass-messaging functionality and can generate lists like other users.

(readonly) Utility :string|number

The Utility pseudo-enum of the main namespace object is used to store various constants for general use in a variety of contexts. The constants of the number data type are related to standardizing edit interval rates and edit delays in cases of rate limiting. Originally, these were housed in a const object in the script-global namespace, though their exclusive use by the MassEdit class instance made their inclusion into main seem like a more sensible placement decision.

The two string data type members are the key name used to store HTML "scenes" (operation interfaces) in the browser's localStorage and the name of the "scene" serving as the first interface built and displayed to the user upon initialization. By convention, this is the "Find and replace" scene, though any scene could have been used.
